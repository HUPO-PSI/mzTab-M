=== Schema and API Specification

The canonical schema definitions for mzTab-M 2.1 live in the `schema/` directory.

==== JSON Schema (`mzTab_2_1-M.json`)

`schema/mzTab_2_1-M.json` is the primary JSON Schema (draft 2020-12) describing all
mzTab-M 2.1 object models.  It is the source of truth for field definitions,
types, validation policies, descriptions, and examples.

==== OpenAPI Specification (`mzTab_2_1-M_openapi.json`)

`schema/mzTab_2_1-M_openapi.json` is a valid OpenAPI 3.1.1 document describing the
mzTab-M REST validation and conversion API.  The 25 shared object model schemas
are referenced externally from `mzTab_2_1-M.json` rather than duplicated, so the
two files stay in sync automatically.

NOTE: The previous OpenAPI YAML file (`mzTab_2_1-M_openapi.yml`) has been superseded
by `mzTab_2_1-M_openapi.json`.  All schema definitions have been consolidated into
the JSON Schema (`mzTab_2_1-M.json`) and the OpenAPI JSON spec references them from
there. The YAML file has been removed.

===== Converting the OpenAPI YAML to JSON

If the YAML spec is updated and needs to be re-converted to JSON, use the provided
helper script:

[source,bash]
----
cd schema
python3 openapi_to_jsonschema.py \
    --input  mzTab_2_1-M_openapi.yml \
    --output mzTab_2_1-M_openapi.json
----

The script extracts `components/schemas` from the OpenAPI YAML, rewrites all
`$ref: '#/components/schemas/X'` pointers to `$ref: '#/$defs/X'`, converts
`x-mztab-example` extension fields to standard JSON Schema `examples` arrays,
and writes a self-contained JSON Schema document.

==== Generating the AsciiDoc Field Reference (`mzTab_2_1-M_schema.adoc`)

A human-readable AsciiDoc reference document listing every mzTab-M field with its
description, type, mandatory status, and example usage can be generated from
`mzTab_2_1-M.json` using the provided script:

[source,bash]
----
cd schema
python3 generate_schema_adoc.py
----

This writes `schema/mzTab_2_1-M_schema.adoc` by default.  Both the input schema
and the output path can be overridden:

[source,bash]
----
python3 generate_schema_adoc.py \
    --schema mzTab_2_1-M.json \
    --output ../specification_documents/mzTab_2_1-M_schema.adoc
----

===== Prerequisites

Python 3.8 or later is required.  No third-party packages are needed for
`generate_schema_adoc.py`.  The `openapi_to_jsonschema.py` conversion script
additionally requires PyYAML:

[source,bash]
----
pip install pyyaml
----

==== Updating the Antora Documentation Partials

The Antora documentation site (`docs/mztabm/`) includes the field reference and
the specification as AsciiDoc _partials_.  These partials are pre-processed
versions of the canonical source files and **must be regenerated** whenever
`schema/mzTab_2_1-M.json` or
`specification_documents/mzTab_format_specification_2_1-M.adoc` change.

Use the `gen-docs.sh` helper script in the repository root:

[source,bash]
----
# Regenerate all documents (schema reference, specification partial, and changes comparison)
./gen-docs.sh

# Regenerate only the schema reference (after JSON schema edits)
./gen-docs.sh --schema

# Regenerate only the specification partial (after spec document edits)
./gen-docs.sh --spec

# Regenerate only the v2.0→v2.1 element comparison document
./gen-docs.sh --changes
----

Each mode produces the following output files:

[cols="1,3",options="header"]
|===
| Flag | Output files

| `--schema`
a|
* `docs/mztabm/modules/developers/partials/mzTab_m_2_1_schema.adoc` (Antora partial)
* `specification_documents/mzTab_m_2_1_schema.adoc` (include target for the standalone spec)

| `--spec`
| `docs/mztabm/modules/developers/partials/mzTab_format_specification.adoc`

| `--changes`
a|
* `specification_documents/mzTab_m_2_0_to_2_1_changes.adoc` (standalone comparison document)
* `docs/mztabm/modules/developers/partials/mzTab_m_2_0_to_2_1_changes.adoc` (Antora partial)
|===

The `--changes` mode requires `compare_adoc_specs.py` (repository root) and compares
`specification_documents/mzTab_m_2_0_schema.adoc` (v2.0 reference) against
`docs/mztabm/modules/developers/partials/mzTab_m_2_1_schema.adoc` (current v2.1 field reference).
Run `--schema` first if the JSON schema was updated before running `--changes`.

After running the script, commit the updated files:

[source,bash]
----
git add docs/mztabm/modules/developers/partials/mzTab_m_2_1_schema.adoc \
        docs/mztabm/modules/developers/partials/mzTab_format_specification.adoc \
        docs/mztabm/modules/developers/partials/mzTab_m_2_0_to_2_1_changes.adoc \
        specification_documents/mzTab_m_2_1_schema.adoc \
        specification_documents/mzTab_m_2_0_to_2_1_changes.adoc
git commit -m "docs: regenerate Antora partials"
git push
----

See the *Documentation Workflow* section in the top-level `README.md` for the
full site-preview and release instructions.

=== Working with AsciiDoc

* Homepage: http://asciidoctor.org/

==== Prerequisites
If you want to edit the AsciiDoc documents (*.adoc) in this directory,
you can use any text editor application available on your platform. 
Alternatively, GitHub also offers native rendering of AsciiDoc.

However, we recommend to use Atom as an editor (https://atom.io/) together 
with the AsciiDoc Preview extension (https://atom.io/packages/asciidoc-preview).

=== Editing / Highlighting AsciiDoc

If you want to edit the document, have a look at the admonitions prefixed with

  IMPORTANT: TODO

They also contain the keyword TODO to make automatic finding of TODO items easier.
Unfortunately, GitHub does not highlight text marked with #highlight me#, so that 
TODOs should be close to the text or section that should be updated.
